# 最小搜索 Agent 需求说明与验收标准

## 1. 项目背景

为了学习 Agent 的 runtime 机制，需要实现一个最小可用的搜索 Agent。该 Agent 只聚焦一个最简单的能力链路：

1. 用户输入问题
2. LLM 判断是否需要搜索
3. 如果需要，则调用搜索工具执行查询
4. LLM 综合搜索结果生成最终答案
5. 如果不需要，则直接回答

本项目的核心目标不是做复杂功能，而是跑通一个现代 Agent 的最小闭环，便于理解 runtime 中的“决策、工具调用、结果回填、最终生成”流程。

## 2. 建设目标

项目需要达到以下目标：

- 实现一个可运行的最小 Agent runtime
- 支持 LLM 基于 API 进行推理与决策
- 支持一个唯一工具：搜索
- 支持“是否搜索”的判断逻辑
- 支持搜索结果回传给 LLM 后生成最终答案
- 支持基础日志，便于观察每一步 runtime 执行过程
- 保持实现简单、清晰、可学习，不追求复杂工程化

## 3. 用户场景

典型使用场景如下：

- 用户输入常识类问题，如“1+1 等于几”，Agent 可直接回答，不调用搜索
- 用户输入需要外部信息的问题，如“帮我查一下某个主题的最新信息”，Agent 判断需要搜索，调用搜索工具后再回答
- 用户希望通过日志看到 Agent 的运行轨迹，理解 runtime 如何工作

## 4. 产品范围

本期范围仅包含：

- 单用户输入
- 单轮问答
- 单个 LLM API 接入
- 单个搜索工具
- 一次或有限次搜索调用
- 最终自然语言答案输出
- 基础运行日志

本期不包含：

- 多轮对话记忆
- 多 Agent 协作
- 复杂任务规划
- RAG 知识库
- 权限体系
- 用户系统
- 前端复杂 UI
- 流式输出不是必须
- 搜索结果排序优化不是重点

## 5. 核心流程

Agent 的运行流程定义如下：

1. 用户提交问题
2. Runtime 将用户问题发送给 LLM
3. LLM 返回决策结果：
   - 直接回答
   - 或发起搜索工具调用
4. 如果触发搜索：
   - Runtime 调用搜索接口
   - 获取搜索结果
   - 将搜索结果回传给 LLM
5. LLM 基于问题和搜索结果生成最终答案
6. Runtime 输出最终答案，并记录执行日志

## 6. 功能需求

### 6.1 用户输入

- 系统应支持用户输入一条自然语言问题
- 输入内容以文本为主
- 不要求支持图片、文件、语音

### 6.2 LLM 决策

- 系统应通过 LLM API 进行推理
- LLM 应能根据用户问题判断是否需要搜索
- LLM 的决策结果应具备明确结构，至少区分：
  - `无需搜索，直接回答`
  - `需要搜索，并给出查询词`

### 6.3 搜索工具

- 系统应提供一个搜索工具接口
- 搜索工具输入为查询词
- 搜索工具输出为结构化结果，建议至少包含：
  - 标题
  - 摘要
  - 链接
- 搜索结果数量可限制为固定条数，例如 Top 5

### 6.4 搜索执行

- 当 LLM 判断需要搜索时，Runtime 应调用搜索工具
- Runtime 应将搜索结果回填给 LLM
- LLM 应基于搜索结果进行总结，而不是机械原样返回

### 6.5 最终回答

- 系统应输出清晰、完整、自然语言形式的最终答案
- 当答案来源于搜索结果时，应尽量体现“基于搜索结果整理”的特征
- 如果搜索结果不足，应允许输出“不确定”或“未找到充分信息”

### 6.6 运行日志

系统应记录最小必要日志，至少包括：

- 用户问题
- 是否触发搜索
- 搜索查询词
- 搜索返回条数
- 最终答案生成完成

日志目标是帮助学习 runtime，不要求复杂监控平台。

## 7. 非功能需求

### 7.1 简单性

- 架构应尽量简单
- 模块边界清晰，便于理解和二次修改

### 7.2 可观察性

- Agent 每一步执行过程应可追踪
- 出现异常时，应能定位是在 LLM 决策、搜索调用还是答案生成阶段

### 7.3 可配置性

至少支持以下配置项：

- LLM API 地址
- LLM API Key
- 模型名称
- 搜索接口地址或搜索服务配置
- 搜索返回条数
- 日志级别

### 7.4 稳定性

- LLM API 调用失败时，系统应返回友好错误信息
- 搜索失败时，系统应提示搜索失败，而不是程序直接崩溃
- 任一外部接口超时时，系统应可中断并返回明确提示

## 8. 建议的模块划分

为了便于学习，建议拆分为以下模块：

- `User Input`：接收用户问题
- `Agent Runtime`：控制整体流程
- `LLM Client`：封装 LLM API 调用
- `Search Tool`：封装搜索能力
- `Prompt / Tool Schema`：定义工具调用规则
- `Logger`：记录执行链路
- `Final Response Renderer`：输出最终结果

## 9. 接口约束建议

### 9.1 LLM 输出格式

建议 LLM 决策阶段输出固定结构，例如：

```json
{
  "need_search": true,
  "query": "最新的 AI Agent runtime"
}
```

如果无需搜索：

```json
{
  "need_search": false,
  "answer": "这是一个常识问题，可以直接回答。"
}
```

### 9.2 搜索结果格式

建议统一为：

```json
[
  {
    "title": "结果标题",
    "snippet": "结果摘要",
    "url": "https://example.com"
  }
]
```

这样 runtime 更容易理解和调试。

## 10. 异常场景

系统需要考虑以下场景：

- LLM 返回格式不合法
- LLM 判断需要搜索，但未给出查询词
- 搜索接口无结果
- 搜索接口超时或失败
- LLM 在总结阶段失败
- 用户输入为空

对以上场景，系统应给出明确提示，不能直接崩溃。

## 11. 最终验收标准

### 11.1 功能验收标准

1. 系统可以接收用户输入的一条文本问题，并正常进入 Agent runtime。
2. 系统已成功接入 LLM API，能够完成一次完整推理调用。
3. 对于明显不需要搜索的问题，LLM 能输出“不搜索”的决策，系统可直接返回答案。
4. 对于明显需要外部信息的问题，LLM 能输出“需要搜索”的决策，并提供可执行查询词。
5. Runtime 在收到搜索决策后，能够成功调用搜索工具并拿到结果。
6. 搜索结果能够被回传给 LLM，LLM 可基于结果生成最终答案。
7. 最终输出对用户可读，不是裸 JSON，也不是原始工具返回。
8. 当搜索无结果时，系统能返回合理提示，而不是报错退出。
9. 当 LLM API 调用失败时，系统能返回明确错误信息。
10. 当搜索接口失败或超时时，系统能返回明确错误信息。

### 11.2 流程验收标准

1. 整体链路必须跑通：用户提问 -> LLM 判断 -> 搜索调用（可选）-> 结果整合 -> 最终回答。
2. 系统可以清楚区分两条路径：
   - 直接回答路径
   - 搜索后回答路径
3. 每次请求都能输出关键日志，至少能看见：
   - 用户输入
   - 是否搜索
   - 搜索词
   - 搜索结果条数
   - 最终完成状态

### 11.3 数据结构验收标准

1. LLM 的决策输出为结构化数据，能被程序稳定解析。
2. 搜索工具返回结果为统一结构，至少包含标题、摘要、链接。
3. Runtime 内部模块之间的数据传递格式清晰、稳定，不依赖临时文本拼接。

### 11.4 可用性验收标准

1. 本地开发环境下能够一键启动或按简单步骤启动。
2. 配置项清晰，至少支持 API Key、模型名、搜索配置。
3. 新开发者可以通过阅读代码和日志，快速理解 runtime 的执行机制。

### 11.5 代码与架构验收标准

1. LLM 调用逻辑与搜索工具逻辑解耦。
2. Runtime 负责流程编排，不直接耦合具体搜索实现细节。
3. 代码结构清晰，核心模块职责明确。
4. 不引入与当前目标无关的复杂设计。

### 11.6 最小测试验收标准

至少覆盖以下用例：

1. 输入常识问题，系统直接回答，不调用搜索。
2. 输入需要外部信息的问题，系统调用搜索后再回答。
3. 搜索返回空结果，系统能友好提示。
4. LLM 接口异常，系统能友好提示。
5. 搜索接口异常，系统能友好提示。

## 12. 补充说明

本项目定位为学习型最小实现，优先保证链路清晰、模块简单、日志可观察。后续如果需要扩展，可以在当前基础上继续增加多轮对话、更多工具、规划能力、流式输出和更完善的测试体系。
